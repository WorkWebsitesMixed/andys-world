Last week you built a model and found problems visually. Today you find problems with measurements. An objective test — actual weights, actual readings, before-and-after data — is what turns a model into development evidence. You also breadboard the switch-and-LED circuit to confirm it works before you commit it to your product.

<Callout type="exam" title="Why this matters" id="why">
  <ExamLink to="project">Component 2</ExamLink> criterion **C4 (development, 15 marks)** rewards objective testing — not opinion. "It looks stable" scores nothing. "It held 4.2 kg without deflecting, but tipped at 5.1 kg when the load was placed asymmetrically, so I added a rear brace and re-tested to 5.5 kg" earns marks. The iteration loop — test → modify → re-test — is what C4 is looking for.
</Callout>

<KeyTerms terms={[
  { term: "Objective testing", definition: "Testing that produces a measurable result — a number, a pass/fail against a specification point — rather than an opinion. 'It held 5 kg without tipping' is objective. 'It seemed stable enough' is not." },
  { term: "Fair test", definition: "A test in which only one variable changes at a time. To test load capacity, you apply a standard weight in a standard position — the same way every time — so results can be compared before and after a design change." },
  { term: "Iteration loop", definition: "The cycle of test → result → design change → re-test. Each loop produces a development point and evidence. One loop is minimum for C4; two or three loops show genuine development." },
  { term: "Breadboarding", definition: "Building a temporary version of an electronic circuit on a solderless breadboard, allowing components to be inserted and removed without soldering. Used to confirm a circuit works before committing it to the permanent product." },
]} />

<Section title="Testing your structure and your circuit" icon="🔬" id="testing">

### Setting up a structural test

Choose a spec point that you can measure — most likely the load specification or a stability condition.

1. **State the test:** "I will apply [X] kg at [position] and observe whether the structure deflects more than [Y] mm or tips."
2. **Set up the fair test:** same weight, same position, same surface each time.
3. **Record the result:** photograph + measurement. Does it pass or fail the spec point?
4. **Design change:** if it fails, record what you changed and re-test.

Common load test: stack weights (textbooks, water bottles) on the critical point of the structure. Measure any deflection at the centre or tip with a rule. Compare before and after a design modification.

### Breadboarding the circuit

The circuit you designed in Term 1 (switch → current-limiting resistor → LED) needs to be built and confirmed working before you integrate it into the product.

Components you need:
- A 9 V battery and holder
- Your chosen switch (toggle, push-to-make, or reed)
- A current-limiting resistor (calculated value from Term 1)
- An LED

Connect them in series on the breadboard. Operate the switch — the LED should light. If it does not: check the LED polarity (long leg to positive), check the connections are in the same rail, check the resistor value is not too high.

</Section>

<WorkedExample title="Testing a cyclist gear organiser — two iterations">

**Test 1 — Load, 4 kg static:** Three 1.5 kg water bottles placed on the horizontal member at height 200 mm. Result: structure held without tipping, deflection at top = 3 mm. Passes the load spec.

**Test 2 — Asymmetric load:** All three bottles placed on one side only. Result: structure tips at 2.7 kg (fails spec — user may load unevenly). Development change: widen the base from 150 mm to 210 mm and add a rear horizontal tie at 50 mm from the floor.

**Re-test after change:** Same asymmetric load to 3.5 kg — no tipping. Passes.

**Circuit test:** Toggle switch → 470Ω resistor → red LED on 9 V supply. LED lit on first attempt. Switch operates cleanly.
</WorkedExample>

<Activity title="Load test + breadboard" time="90 min" grouping="Individual (technician supervises breadboard station)" materials={["Model from Week 4", "Weights (water bottles or book set)", "Ruler / tape measure", "Breadboard kit: battery, switch, resistor, LED", "Camera (phone)"]}>

**Load or stability test (0–45 min):**
1. State your test: spec point, weight, position, pass/fail criterion.
2. Set up and run the test. Photograph the result.
3. Record: does it pass or fail? By how much?
4. If it fails, make one design change and re-test. Record before and after.

**Breadboard the circuit (45–75 min):**
5. Using your Term 1 circuit diagram and resistor calculation, build the circuit on a breadboard.
6. Operate the switch. Does the LED light?
7. If not: check LED polarity, check connections, check resistor is not open-circuit.
8. Note which switch type you chose for your product and confirm it operates as expected in the physical test.

**Record and tidy (75–90 min):**
9. Write up your test-results table: spec point / test condition / result / pass? / change made.
10. Note any changes the results call for in your final product.
</Activity>

<Support title="Test not giving clear results?">

**Result is ambiguous — hard to say pass or fail?** Go back to your specification. The spec point should have a measurable standard. If it says "must feel stable", revise it: "must not tip when 5 kg is applied at the outermost point". Now you have a pass/fail.

**Circuit not lighting?** Work systematically: (1) Is the battery charged? Check with a voltmeter or test light. (2) Is the LED polarity correct? Flip it. (3) Are all jumper wires in the same column of the breadboard (not bridging rows accidentally)? (4) Is the resistor value reasonable — not too high (LED dim or off)?

**Can't fit the breadboard test into one session?** Prioritise the structural test — it gives C4 development evidence directly. The circuit test can run in parallel while the structure re-test cure time allows.
</Support>

<Section title="Homework" icon="🏠" id="homework">

**Home task:** Write up your test-results table neatly for the folder. For each test, include: spec point tested, method, result (with measurement), pass/fail, and any change made. Photograph the model at each stage if you have not already done so.

<Extension title="Second structural test">
Design and run a second structural test on a different specification point — for example, test rigidity (does the frame rack under a sideways push?) or stability (does the base slide on a smooth surface?). Record the method, result, and any change. Add it to your C4 development pages.
</Extension>
